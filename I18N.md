# Internationalization (i18n)

This template ships with [`next-intl`](https://next-intl-docs.vercel.app/) wired
up, but only the `/about` page has been migrated as a demonstration. The rest
of the public site is still single-locale (Chinese). Use this guide to migrate
additional pages incrementally.

## Current state

| Path | Status |
| --- | --- |
| `/` | Redirects to `/zh` |
| `/zh`, `/en` | Migrated homepage (`src/app/[locale]/page.tsx`) |
| `/about` | Redirects to `/zh/about` |
| `/zh/about`, `/en/about` | Migrated about page |
| `Navbar`, `Footer`, `LocaleSwitcher` | Use `useTranslations` and i18n `Link` |
| `/archives`, `/posts/[id]`, `/admin/*`, `/api/*` | Single-locale (Chinese), not migrated |

The `proxy.ts` matcher is scoped to `/(zh|en)/:path*`, so non-i18n routes are
not affected by locale rewriting.

The locale switcher in the Navbar swaps between `zh` and `en` while keeping
the user on the equivalent localized route. Switching to `en` while viewing
an unmigrated route (e.g. `/archives`) keeps you on the Chinese version —
that's the trade-off of incremental migration.

## Layout

```
src/i18n/
├── routing.ts        # locales, defaultLocale, prefix mode
├── request.ts        # loads messages for the active locale
└── navigation.ts     # locale-aware <Link>, useRouter, redirect
src/proxy.ts          # next-intl middleware (Next 16 calls it "proxy")
src/app/[locale]/
├── layout.tsx        # wraps children in NextIntlClientProvider
└── about/page.tsx    # demo page
messages/
├── zh.json
└── en.json
```

## Migrating a page

1. **Move the page** from `src/app/<route>/page.tsx` to
   `src/app/[locale]/<route>/page.tsx`.
2. **Add `setRequestLocale`** at the top of the server component so static
   rendering knows the locale:
   ```tsx
   import { getTranslations, setRequestLocale } from 'next-intl/server';

   export default async function Page({
     params,
   }: {
     params: Promise<{ locale: string }>;
   }) {
     const { locale } = await params;
     setRequestLocale(locale);
     const t = await getTranslations('myPage');
     return <h1>{t('title')}</h1>;
   }
   ```
3. **Add `generateStaticParams`** to pre-render every locale:
   ```tsx
   import { routing } from '@/i18n/routing';
   export function generateStaticParams() {
     return routing.locales.map((locale) => ({ locale }));
   }
   ```
4. **Replace hardcoded strings** with `t('...')` calls and add the keys to
   both `messages/zh.json` and `messages/en.json`.
5. **Update `<Link>` and `router.push`** in client components to import from
   `@/i18n/navigation` instead of `next/link` / `next/navigation`. They
   automatically prefix the current locale.
6. **Delete or redirect the legacy route** at `src/app/<route>/page.tsx`.
   Either remove the file (and any links to `/about`) or replace it with a
   `redirect('/zh/<route>')` so old URLs keep resolving.

## Locale switcher

Not implemented yet. To add one (typically in `Navbar`), use
`useLocale()` and the `Link` from `@/i18n/navigation`:

```tsx
'use client';
import { useLocale } from 'next-intl';
import { Link, usePathname } from '@/i18n/navigation';

export function LocaleSwitcher() {
  const locale = useLocale();
  const pathname = usePathname();
  const next = locale === 'zh' ? 'en' : 'zh';
  return <Link href={pathname} locale={next}>{next.toUpperCase()}</Link>;
}
```

## What still needs to be done for a fully bilingual site

- [ ] Migrate `/`, `/archives`, `/posts/[id]` under `[locale]/`.
- [ ] Migrate shared components (`Navbar`, `Footer`, `SearchModal`,
      `CommentSection`, `Breadcrumb`) to use `useTranslations`.
- [ ] Update `sitemap.ts`, `robots.ts`, `rss.xml`, and both
      `opengraph-image.tsx` files to emit per-locale entries.
- [ ] Add a locale switcher to `Navbar`.
- [ ] Decide how `Post` content itself should be localized — adding a
      `locale` column on `Post` and querying by it is the simplest path;
      modeling translations as a separate table is the flexible one.
- [ ] Once everything is migrated, switch `routing.ts` to
      `localePrefix: 'as-needed'` so `/about` (no prefix) means Chinese
      and `/en/about` is English. Then expand the matcher in `proxy.ts`
      to cover all public routes.

## Adding a new locale

1. Add the code (e.g., `'ja'`) to `routing.ts → locales`.
2. Create `messages/ja.json` with the same key shape as `zh.json`.
3. Update the matcher in `proxy.ts`: `'/(zh|en|ja)/:path*'`.
